# AGENTS.md

このリポジトリは、macOS の開発環境を再現可能な形で管理するための dotfiles リポジトリです。

AI エージェントがこのリポジトリを編集する場合は、単に「今動く設定」を追加するのではなく、再セットアップ・別マシンへの移行・CI・ドライランでの検証を前提にしてください。

## 基本方針

このリポジトリでは、次の原則を優先します。

1. **べき等性**

   * セットアップは何度実行しても安全であること。
   * 初回だけ成功する処理ではなく、同じ状態に収束する処理にすること。

2. **安全なドライラン**

   * `./setup.sh --dry-run` で副作用なく実行予定を確認できること。
   * ファイル作成・削除・リンク作成・インストールなどの状態変更は、ドライランで実行されないようにすること。

3. **管理責務の分離**

   * Homebrew、mise、opam、npm などの役割を混在させないこと。
   * 同じツールを複数の仕組みで管理しないこと。
   * dotfiles で管理するものと、プロジェクト固有・マシン固有のものを分けること。

4. **秘密情報を含めない**

   * API キー、トークン、パスワード、個人用の環境変数を Git 管理対象に含めないこと。
   * 秘密情報は Git 管理外の hidden 用設定に分離すること。

5. **将来の自分が読めること**

   * その場しのぎではなく、後から見ても意図が分かる変更にすること。
   * コメントは「何をしているか」より「なぜそうしているか」を説明するために使うこと。

## AI エージェントへの指示

* 既存の方針を尊重し、無関係なリファクタリングを同時に行わない。
* 変更の影響範囲を小さく保つ。
* OS やマシン固有の仮定をできるだけ避ける。
* 破壊的な操作、設定上書き、削除、アンインストールは慎重に扱う。
* セットアップ処理を変更した場合は、必ずドライランで確認できる状態を保つ。
* CI で通ることだけでなく、実際のユーザー環境で繰り返し実行できることを重視する。

## setup.sh のルール

`setup.sh` はこのリポジトリの中心的なセットアップ入口です。

システム状態を変更するコマンドは、原則として直接実行せず、ドライラン対応のラッパー（例: `execute_cmd`, `run_and_log`, `run_cmd`）を経由してください。

対象例:

* `mkdir`
* `ln`
* `mv`
* `rm`
* `curl`
* `git clone`
* ファイルへの追記
* 設定ファイルの生成
* パッケージインストール
* 権限変更

例:

```bash
execute_cmd "mkdir -p \"$HOME/.config/example\""
execute_cmd "ln -snf \"$DOTFILES_DIR/.config/example/config\" \"$HOME/.config/example/config\""
```

シンボリックリンクは、原則として `ln -snf` を使い、安全に上書きできる形にしてください。

ファイルへ追記する場合は、同じ内容が重複しないようにしてください。

```bash
grep -q 'some setting' "$file" || echo 'some setting' >> "$file"
```

`set -e` が有効な環境では、`grep` などが仕様上非ゼロを返す場合に注意してください。

## パッケージ管理

### Homebrew

Homebrew は、macOS アプリ、CLI ツール、Cask、必要に応じて VS Code 拡張などを管理します。

`Brewfile` を更新するときは、標準の `brew bundle dump` を直接使わず、このリポジトリで用意している `brew-dump` を使ってください。

```bash
brew install <package-name>
cd ~/dotfiles
brew-dump
```

理由は、mise や npm など別の仕組みで管理しているツールが `Brewfile` に混入するのを避けるためです。

### mise

mise は、Node.js、Python、Go、Java などの言語ランタイムや開発ツールの管理に使います。

Homebrew で入れるべきものと mise で入れるべきものを混在させないでください。

### opam

opam は OCaml 環境の管理に使います。

OCaml 関連の変更では、Homebrew や mise との責務の重複に注意してください。

## 秘密情報・ローカル設定

Git 管理対象の設定ファイルに秘密情報を書かないでください。

秘密情報やマシン固有の設定は、Git 管理外の hidden 用ファイルに分離してください。

例:

```text
~/.config/zsh/hidden/
```

## CI 方針

GitHub Actions では、ユーザーの実環境とは異なるパス・権限・初期状態で実行されます。

そのため、次に注意してください。

* `DOTFILES_DIR` を `~/dotfiles` に固定しない。
* CI の `$PWD` をリポジトリルートとして扱えるようにする。
* `set -e` 環境で非ゼロ終了する可能性のある処理を安全に扱う。
* CI 専用の特殊処理を増やしすぎず、実環境での自然な挙動を優先する。

## 変更後の確認

`setup.sh` や管理設定を変更した後は、まずドライランで確認してください。

```bash
./setup.sh --dry-run
```

確認する内容:

* パスが正しく展開されているか
* 予期しないファイルを変更しようとしていないか
* 破壊的操作が直接実行されていないか
* 同じ処理が重複していないか

問題なければ本番実行します。

```bash
./setup.sh
```

## 判断に迷ったときの優先順位

迷った場合は、次の順に優先してください。

1. 秘密情報を漏らさない
2. ユーザーの環境を壊さない
3. ドライランで安全に確認できる
4. 何度実行しても同じ状態に収束する
5. 管理責務を重複させない
6. 将来の自分が理解できる
7. 便利さを追加する

便利さよりも、安全性・再現性・見通しの良さを優先してください。
