# Roadmap

このファイルは、今後検討したい改善候補をまとめる場所です。

ここにある内容は、確定した方針や対応予定ではありません。public repository に置くため、個人情報、内部事情、具体的すぎる弱点、期限の約束は書きません。

## Windows / WSL 対応

macOS 以外の環境でも、一部の shell / editor 設定を再利用できるか検討します。

検討したいこと:

- macOS 専用設定との分離
- Homebrew 前提の処理をどう扱うか
- `setup.sh` を分けるか、OS 判定を入れるか
- WSL で管理する範囲をどこまでにするか

## Nix ベース環境への移行

できれば Nix / nix-darwin / Home Manager を使い、より再現性の高い環境構築へ移行できるか検討します。

検討したいこと:

- 既存の `setup.sh` と並走するか、段階的に置き換えるか
- macOS アプリ、CLI ツール、言語ランタイムを Nix 側でどこまで管理するか
- Homebrew、mise、opam で管理しているものを移行時にどう扱うか
- Nix を使わない環境向けの逃げ道を残すか

## setup.sh の安全性改善

`./setup.sh --dry-run` をより安全で読みやすくし、実行予定を信頼しやすくします。

検討したいこと:

- dry-run 表示で不要な評価が起きないようにする
- 状態変更コマンドのラッパーを整理する
- 失敗時のログとエラーメッセージを分かりやすくする

## symlink のべき等性改善

既に正しいリンクがある場合に、不要なバックアップや変更予定が出ないようにします。

検討したいこと:

- 既存の正しい symlink を検出してスキップする
- 通常ファイル、ディレクトリ、symlink の扱いを明確にする
- dry-run の出力を「作成」「更新」「スキップ」に分ける

## CI の軽量化

GitHub Actions で、実環境に近い確認を保ちつつ、重い処理を減らせるか検討します。

検討したいこと:

- dry-run と構文チェックの役割を分ける
- Homebrew / mise / opam の実インストール確認をどこまで CI で行うか
- CI 専用の特殊処理を増やしすぎない構成

## CI 検証方針の二層化

PR で毎回確認する内容と、手動または必要時だけ確認する内容を分けられるか検討します。

検討したいこと:

- PR CI では設定ファイルの構文、dry-run、本番に近い軽量セットアップを確認する
- 手動 CI では Homebrew / mise / opam を含む、より実環境に近いセットアップ確認を行う
- 重い処理をスキップした場合でも、どの確認が省略されたかログから分かるようにする
- skip flag が増えすぎた場合に、用途別の実行モードとして整理する

## setup.sh の段階分割

`setup.sh` が担う処理を段階ごとに分け、CI やローカル検証で必要な部分だけ実行しやすくできるか検討します。

検討したいこと:

- ディレクトリ作成、シンボリックリンク作成、Homebrew、mise、opam を段階として整理する
- 各段階を個別に dry-run できるようにする
- ローカル通常実行は従来どおり一括で完了する入口として残す
- 段階分割によって、べき等性やログ出力が分かりにくくならないようにする

## 管理対象の棚卸し

dotfiles で管理するもの、ツールごとに管理するもの、マシン固有として管理しないものを定期的に見直せる形にします。

検討したいこと:

- Homebrew、mise、opam、VS Code、Codex、shell 設定の責務境界を明文化する
- 既存の `Brewfile` と mise 設定に重複管理がないか確認する
- public repository に置ける設定と hidden 設定に分ける基準を見直す
- README / docs / AGENTS.md の説明が実装とずれていないか確認する
